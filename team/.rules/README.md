# Coding Rules Directory

This directory manages coding rules by project type.

---

## 📂 Directory Structure

```
.rules/
├── README.md                        # This file
├── general-coding-rules.md          # General coding principles (DRY, SOLID, etc.)
├── _verified/                       # ⭐ Human-verified rules (highest priority)
│   └── web-fullstack/
│       ├── be-coding-rules.md       # FastAPI (Python) - Verified
│       └── fe-coding-rules.md       # Next.js (TypeScript) - Verified
└── _cache/                          # 🤖 Auto-generated rules by Stack Initializer Agent
    └── (auto-generated on project start)
```

---

## 🔍 Priority Order

When the coding agent looks for coding rules, it checks in this order:

1. **`_verified/{project_type}/{framework}-{language}.md`** - Human-verified rules
   - **Exists**: Use this rule (highest trust)
   - **Does not exist**: Check #2

2. **`_cache/{project_type}/{framework}-{language}.md`** - Auto-generated rules
   - **Exists + within 24 hours**: Use this rule
   - **Exists + over 24 hours**: Ask user whether to regenerate
   - **Does not exist**: Proceed to #3

3. **Real-time generation** - Call Stack Initializer Agent
   - Analyze official documentation and best practices
   - Generate new rule in `.rules/_cache/`
   - Cache for 24 hours

---

## ⭐ Promoting Rules to Verified

Promote auto-generated rules to verified status:

```bash
# 1. Check auto-generated rule
cat .rules/_cache/cli-tool/cobra-go.md

# 2. Make necessary edits

# 3. Move to _verified
mkdir -p .rules/_verified/cli-tool
mv .rules/_cache/cli-tool/cobra-go.md .rules/_verified/cli-tool/

# 4. Update .project-config.json
# "coding_rules_status": "verified"
```

## 🤖 Auto-Generated Rule Characteristics

Rules generated by Stack Initializer Agent include:

- **Project Structure**: Directory layout, file organization
- **Architecture Patterns**: MVC, Layered, Clean Architecture, etc.
- **Naming Conventions**: File names, variable names, function names, class names
- **Security Guide**: Input validation, secrets management, OWASP responses
- **Testing Strategy**: Frameworks, directory structure, coverage goals
- **Dependency Management**: Package managers, version control
- **Error Handling**: Exception handling strategy, logging
- **Performance Optimization**: Framework-specific optimization points
- **References**: Official documentation links, popular project examples

## 🔄 Cache Refresh

Files under `.rules/_cache/` are cached for 24 hours.

### Manual Refresh

```bash
# Regenerate specific rule
rm .rules/_cache/cli-tool/cobra-go.md
bash scripts/run-agent.sh stack-initializer --config .project-config.json
```

### Automatic Refresh

Stack Initializer Agent checks cache file creation time:
- Within 24 hours: Use existing file
- Over 24 hours: Ask whether to regenerate

## 🛠️ Example: CLI Tool (Go Cobra)

### Initial State (no verified rule)

```bash
bash scripts/init-project.sh --type cli-tool --language go --framework cobra --name my-cli
```

Stack Initializer Agent:
1. Check `.rules/_verified/cli-tool/cobra-go.md` → Does not exist
2. Check `.rules/_cache/cli-tool/cobra-go.md` → Does not exist
3. Analyze official documentation → Generate `cobra-go.md`
4. Save to `.rules/_cache/cli-tool/cobra-go.md`

### After Verification and Promotion

```bash
# Edit and verify
vi .rules/_cache/cli-tool/cobra-go.md

# Promote
mkdir -p .rules/_verified/cli-tool
mv .rules/_cache/cli-tool/cobra-go.md .rules/_verified/cli-tool/
```

### Next Project (same stack)

```bash
bash scripts/init-project.sh --type cli-tool --language go --framework cobra --name another-cli
```

Stack Initializer Agent:
1. Check `.rules/_verified/cli-tool/cobra-go.md` → **Exists!**
2. Skip auto-generation, use verified rule
